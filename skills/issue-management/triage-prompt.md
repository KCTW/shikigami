# Issue Triage — 分類標準與留言模板

本文件由 `issue-management` Skill 的 Triage 子流程使用。定義 issue 分類規則與自動留言模板。

---

## 分類規則

依照以下順序判定 issue 類型。若同時符合多個類型，以**最先匹配**的為準。

### 1. Bug（`bug` label）

**判定條件**（符合任一）：
- 描述異常行為或錯誤結果
- 包含錯誤訊息、stack trace、錯誤截圖
- 使用「不能」「失敗」「crash」「error」「broken」等關鍵詞
- 描述「預期 A 但得到 B」的行為差異

**必要資訊**：重現步驟、預期行為、實際行為、環境資訊

### 2. Feature Request（`feature-request` label）

**判定條件**（符合任一）：
- 期望新功能或行為變更
- 使用「希望」「建議」「可以加」「would be nice」等詞彙
- 描述目前缺少的能力

**必要資訊**：使用場景、預期行為、驗收標準

### 3. Question（`question` label）

**判定條件**（符合任一）：
- 詢問使用方式或配置方法
- 使用「怎麼」「如何」「how to」「is it possible」等問句
- 不涉及異常行為，純粹尋求說明

### 4. Documentation（`documentation` label）

**判定條件**（符合任一）：
- 指出文件錯誤、過時資訊、缺少說明
- 建議改善文件內容
- README、CHANGELOG、API docs 相關

### 5. Invalid（`invalid` label）

**判定條件**（符合任一）：
- 無法理解內容（亂碼、無意義文字）
- 明顯非本專案範圍
- 垃圾訊息或廣告

## 留言模板

### Bug — 缺少重現步驟

```markdown
感謝回報！為了幫助我們排查這個問題，請補充以下資訊：

1. **重現步驟**：請描述如何觸發這個問題（步驟 1、2、3...）
2. **預期行為**：你期望看到什麼結果？
3. **實際行為**：實際發生了什麼？
4. **環境資訊**：OS、相關工具版本

補充資訊後我們會儘快處理，謝謝！
```

### Feature Request — 缺少使用場景

```markdown
感謝你的建議！為了更好地評估這個功能，請補充：

1. **使用場景**：在什麼情境下你會需要這個功能？
2. **目前的替代方案**：你現在是如何處理的？
3. **預期行為**：這個功能具體應該怎麼運作？

這些資訊會幫助我們在 Backlog 中正確評估優先級。
```

### Question — 引導至文件

```markdown
你好！感謝提問。

[根據問題內容提供簡短回答或指向相關文件]

如果還有其他問題，歡迎繼續討論。
```

### 確認收到

```markdown
感謝回報，已收到！團隊將評估處理。
```

### 已修復通知

```markdown
這個問題已在 PR #[N] 中修復，將於下次版本釋出。感謝回報！
```

### Won't Fix

```markdown
感謝回報。經團隊評估，目前不打算處理這個項目，原因如下：

[具體說明原因]

如果你有不同看法或新的資訊，歡迎繼續討論。
```

### 已轉入 Backlog

```markdown
感謝提出！這個需求已轉入產品 Backlog，將在後續的 Sprint 規劃中評估排入。
```
